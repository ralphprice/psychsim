# PsychSim — Master Design & Build Document, Part 4 (Supplements & Addenda, continued)

*Sealed, versioned supplement to `PsychSim_MASTER.md` and Parts 1–3. Not updatable once distributed —
further guidance becomes a new Part. Section numbering continues the global sequence (Parts 2 = S1–S2,
Part 3 = S3–S6, this Part = S7–S8).*

This Part carries two things: the **pre-8b.4 phenomena-review ruling** — build the executive-control

## S7. Pre-8b.4 phenomena review — ruling: build the executive-control development arm

### S7.1 On the result (it is the right kind of mixed)

The 8b.3-era divergence probe returned a mixed result, honestly surfaced, and the parts that emerged
are the parts that are hardest to fake:
- the **punishment-learning deficit** graded in the throttle with inversion at full throttle — the
  single most diagnostic CU signature (passive-avoidance failure), from a numeric gain on circuit IDs
  with the outcome measured;
- the **reads-but-doesn't-feel dissociation** — the thing v8 was built for, appearing as predicted;
- **half the differential-susceptibility contrast** (typical internalises under adversity, fearless
  doesn't).

Three grounded phenomena from meaning-blind mechanism. That is pre-registration-quality emergence, not
demonstration. It should not be undersold.

### S7.2 The ruling: yes, build the arm — and why it is not forcing

**Build the executive-control development arm, then re-run the divergence test, before any 8b.4 talk.**
The reason the current test cannot show divergence is not a bug — it **measured the wrong variable**.
Affective/threat tone is the CU *root*; the sophropath/psychopath split is a *behavioural outcome*
downstream of how that root is **regulated by executive control**. Measuring affective tone and looking
for outcome divergence is a category mismatch — they can't diverge, because the variable that separates
the two outcomes (control over behaviour) is neither in what was measured nor in what was built.
Building the arm **completes the causal chain the observer is meant to read at its end, not its
middle.** That is legitimate.

### S7.3 The constraint that keeps it honest (the whole game)

There are two ways to build the arm; only one is allowed.

- **Forbidden:** wire the environment to executive-control strength *and* the throttle to the affective
  root such that "warm → control → adaptive" and "harsh → disinhibition → antisocial" are effectively
  hardcoded paths. If the wiring guarantees the divergence, the answer is coded and the result is
  worthless — worse than the honest negative.
- **Required:** the environment enters **only** as perturbation patterns (as it already does); those
  perturbations drive the executive circuits' activity; the **meaning-blind plasticity** sculpts their
  developed strength from that activity; behaviour selection reads the **result**. Then test whether
  warm-firm-throttled and harsh-throttled children diverge on the observer's behavioural metrics —
  **without ever having told the substrate that warm should yield control or harsh disinhibition.**

The test you must be able to pass afterward: **could someone read the code and find the line where the
environment is mapped to an outcome?** If yes, rip it out. The divergence must survive the same S5.5
checkpoints as everything else — environment as perturbation, development as plasticity forward,
outcome measured, nothing steered.

### S7.4 The control condition — the interaction is the finding

When you build the arm, **also run the intact (un-throttled) child through the same warm/harsh
environments.** The claim is not "throttled children diverge by environment" — it is that the
**interaction** is specific: the fearless child's outcome should be *more* environment-dependent than
the typical child's (that is the *differential*-susceptibility structure, which S7.1's point-3 already
gestures at). If the intact child diverges just as much, you have a general environment effect, not the
psychopathy-specific one. **The interaction is the result, not the main effect.** So the minimal design
is a 2×2: {throttled, intact} × {warm-firm, harsh-inconsistent}, outcome measured.

### S7.5 What this ruling is and isn't

This is **not** "the current result is sufficient," and **not** "accept the honest negative" yet. The
current result is insufficient *for the divergence claim specifically* — for a defensible reason (the
arm isn't built), not a fatal one. **The negative isn't earned until the arm exists and the divergence
still fails to appear.** Build the arm honestly, run the 2×2, measure the behavioural/CU outcome, and
**hold before 8b.4.** Sync it, and the design session reviews exactly one thing first: whether the
environment→outcome path is emergent or wired. If clean, the divergence result — positive or negative —
is publishable either way, and *then* the legacy-retirement decision is taken. Legacy stays untouched
until that review.

## S8. The scan controller & permutation search (the study's search mode)

Sequenced **after** S7: build this on a single-character divergence loop that has already passed the
honesty review — not before. Building the search engine before the thing it searches is validated means
scanning with an unchecked instrument.

### S8.1 Framing — an optional mode, not a replacement

The plugin is a **manual slider panel first**: a panel of throttles (0–100) per sim character, one per
throttleable circuit/connection in the affective/empathy network (v8). **Manual mode is the default and
primary use** — you set the throttles from what the research suggests (hypothesis-driven), and observe
what develops. **Auto mode is a feature you actively select**; it hands the same sliders to a search
algorithm. Same substrate, same observer read-out, same honesty wall — the only thing that changes is
*who sets the sliders*, you or the search. Build implication: the manual path and the observer read-out
must work and be trusted first, because **auto is just an automated caller of the same single-character
loop.**

### S8.2 The two auto objectives

- **Search-for-effect** — find throttle combinations that produce a strong **measured** CU/behavioural
  signature. Exploratory; generates candidate hypotheses.
- **Search-for-match-to-field-data** — find the throttle combination whose *developed simulated
  profile* matches an **external, held-out** real-world pattern (a clinical CU profile, a known
  passive-avoidance deficit magnitude, an epidemiological signature). This is **independent
  corroboration**: if the sim, from a meaning-blind circuit disruption plus development, independently
  lands on a configuration that reproduces field data it was never told about, that is convergent
  evidence for the underlying mechanism — the model earning credibility by *rediscovering* reality
  rather than being fitted to it.

### S8.3 The found-not-fitted constraint (the honesty core of auto)

The objective function is where the answer can sneak back in. Rules:
- **Fitness is a MEASURED signature** (an observer read-out) or a **distance to an external held-out
  field pattern** — **never** closeness to a target profile you drew. Maximise *how strongly a real CU
  signature emerges* / *how well the developed profile matches held-out data* — which combination does
  that is the discovery.
- **For search-for-match, the substrate stays FIXED and grounded; ONLY the throttle settings vary; the
  field data is external and never used to adjust the mechanism.** A hit is corroboration **because
  nothing about the mechanism moved to achieve it.** Tuning substrate weights to make the match appear
  would be curve-fitting — worthless.
- **The grep test:** could someone find where a desired *outcome* enters the objective? A measured
  read-out or an external field-pattern distance is clean; a hand-drawn target profile is not.

### S8.4 The search algorithm — coarse-to-fine (mandatory)

The affective network is ~20+ throttleable nodes plus connections, so even the **binary** corners are
2^20+ ≈ millions — un-brute-forceable. An honing algorithm is required from the first run:
- **Phase 1 — coarse screen:** single sliders and low-order combinations at **0/100 only**, to find
  which circuits matter *at all* (most won't).
- **Phase 2 — focused search:** relax to **graded** values (e.g. 25/50/75) **only on the handful of
  levers the screen flagged.** Never sweep a continuous grid over 20 dimensions — only over the 3–4
  that survived. (Method — evolutionary / Bayesian-optimisation / bandit over the throttle vector,
  guided by the measured objective — is the implementer's choice; the invariants are coarse-to-fine and
  a measured objective.)

### S8.5 Parallel instances — independent substrates in one world

- Each character is an **independent substrate instance** carrying its own slider settings. Parallelism
  is honest **precisely because they do not share substrate state**; the shared spawn world gives them
  something to interact with and be measured in, but **substrates must not bleed**, or the throttle
  effect is confounded with cross-contamination.
- **Ecological validity, not just speed:** proto-psychopaths/sophropaths may be ~4% of a population, so
  a large spawn *should* contain several — parallelism reflects reality as well as accelerating the
  scan.
- Architecture: **N independent substrate instances, one shared spawn world.**

### S8.6 The scan controller (the new architectural piece)

A **scan controller / experiment manager**, distinct from the substrate engine and the observer
read-out (nothing in the build does this yet). Four jobs:
1. Hold the **permutation queue and search state** (combinations tried, scores).
2. **Allocate** permutations across available character slots, and **rotate** the queue through them —
   even when there is only one slot.
3. **Collect** each character's measured profile and feed it to the search algorithm to pick the next
   batch.
4. **Log everything** for provenance (settings, developed profile, score, RNG seed).

### S8.7 Two cautions

- **Scaffold × search interact dangerously.** Every throttle result depends on the substrate's
  uncalibrated weights/time-constants. The scan will happily find combinations that produce strong
  signatures **because of a particular scaffold value, not because of neuroscience — and it cannot tell
  the difference.** So a combination the scan flags as "best" is a **hypothesis to check for
  robustness** (does it survive perturbing the scaffold?), **not a finding.** The scan generates
  candidates; it does not validate them. Build that expectation in from the start, or the whole thing
  becomes an elaborate way to overfit the scaffold.
- **This is real infrastructure, sequenced last.** The controller, search algorithm, parallel-instance
  harness, and results store are a bigger build than the pre-8b.4 question — and none of it is that
  question. Order: **S7 (executive-control arm) → phenomena review → 8b.4 decision → then S8.**

### S8.8 Sequencing summary

1. Build the executive-control arm to S7's constraint; run the 2×2; measure; **hold for review.**
2. Design-session reviews environment→outcome (emergent vs wired); take the 8b.4 decision.
3. **Only then** build the S8 scan controller, on the now-trusted single-character loop — manual mode
   first, auto mode (both objectives) as the selectable layer over it.
